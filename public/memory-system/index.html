State DB Memory System for AI Agents

A persistent, automated memory layer for AI agents that lose context between sessions.

Built for OpenClaw. Solves the #1 problem with long-running AI assistants: they forget everything after context compaction.

The Problem

AI agents running on LLMs lose context in three ways:

1. Context compaction — when the conversation gets too long, the system compresses it and the agent loses detail

2. Session boundaries — cron jobs, scheduled tasks, and new conversations start with zero context

3. The "fresh start" problem — the agent asks "what are we working on?" when it should already know

Every attempt to fix this with "just write notes to a file" fails because the agent forgets to write the notes. The same memory loss that creates the problem prevents the solution.

The Solution

Separate the note-taker from the worker.

A lightweight cron job (the "court stenographer") runs every 30 minutes, reads the recent conversation transcript, extracts structured state entries, and writes them to a SQLite database. The worker agent doesn't need to remember to take notes — someone else is always doing it.

After context loss, the agent queries the database instead of reloading full conversation logs. Recovery takes milliseconds and costs ~1K tokens instead of 70K+.

Setup

1. Create the Database


-- Create for each agent that needs persistent memory
-- e.g., ~/.openclaw/workspace/memory/state.db (main agent)
-- e.g., ~/.openclaw/agents/hodl/workspace/memory/state.db (sub-agent)

CREATE TABLE state (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at TEXT DEFAULT (datetime('now', 'localtime')),
  category TEXT NOT NULL,
  domain TEXT DEFAULT 'GENERAL',
  summary TEXT NOT NULL,
  detail TEXT,
  status TEXT DEFAULT 'active',
  session_date TEXT DEFAULT (date('now', 'localtime')),
  superseded_by INTEGER REFERENCES state(id)
);

CREATE TABLE watermarks (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TEXT DEFAULT (datetime('now', 'localtime'))
);

-- Indexes for fast queries
CREATE INDEX idx_state_status ON state(status);
CREATE INDEX idx_state_category ON state(category);
CREATE INDEX idx_state_date ON state(session_date);
CREATE INDEX idx_state_active ON state(status, id DESC);

-- Views for common queries
CREATE VIEW v_active_state AS
SELECT id, created_at, category, domain, summary, detail
FROM state WHERE status = 'active' ORDER BY id DESC LIMIT 50;

CREATE VIEW v_rejected AS
SELECT id, created_at, summary, detail
FROM state WHERE category = 'rejected' AND status = 'active' ORDER BY id DESC;

CREATE VIEW v_today AS
SELECT id, created_at, category, domain, summary, status
FROM state WHERE session_date = date('now', 'localtime') ORDER BY id DESC;

2. Create the Extraction Cron

This is the "court stenographer" — runs every 30 minutes during active hours, reads recent conversation, extracts state changes.


openclaw cron add \
  --name "State Extractor" \
  --cron "/30 8-23   " \
  --tz "America/New_York" \
  --agent main \
  --session isolated \
  --model sonnet \
  --timeout-seconds 60 \
  --no-deliver \
  --message 'STATE EXTRACTION — Automated context capture

You are a state extractor. Read recent conversation and insert structured state entries.

1. Check watermark:
   sqlite3 memory/state.db "SELECT value FROM watermarks WHERE key='\''last_msg'\'';"

2. Read recent messages:
   sessions_history(sessionKey="YOUR_SESSION_KEY", limit=20, includeTools=false)

3. For each message since watermark, extract:
   - DECISIONS: User approved/rejected/decided something
   - TASKS: Something started, completed, or assigned
   - REJECTIONS: User said no (CRITICAL — prevents re-suggesting)
   - AGENT STATE: Agent status changed
   - CONTEXT: Mood, energy, priorities

4. Insert findings:
   sqlite3 memory/state.db "INSERT INTO state (category, domain, summary, detail) VALUES (...);"

5. Update watermark:
   sqlite3 memory/state.db "INSERT OR REPLACE INTO watermarks (key, value) VALUES ('\''last_msg'\'', '\''MSG_ID'\'');"

RULES:
- Extract ONLY clear state changes. Not every message.
- One-line summaries. Detail under 200 chars.
- If nothing meaningful happened, just update watermark.
- NEVER modify existing rows except to mark done/superseded.'

Key design decisions:

  • Sonnet, not Haiku — needs to understand nuance (what's a "rejection" vs a "preference")
  • Every 30 min, not every 5 — balances cost vs freshness. Worst case: 30 min gap after compaction
  • 8 AM - 11 PM only — overnight is cron output, not decisions. No value in extracting.
  • Watermark-based — if a run fails, next run picks up from where the last success left off. Gaps auto-backfill.
  • isolated session — doesn't pollute the main conversation context
  • no-deliver — silent operation, no output to user

3. Configure Context Recovery (Tiered)

Add to your agent's system prompt or AGENTS.md:


## Context Recovery After Compaction

Tier 0 — State DB (TRY FIRST — ~1K tokens):
sqlite3 memory/state.db -header -column "SELECT * FROM v_active_state;"
sqlite3 memory/state.db -header -column "SELECT * FROM v_rejected;"

Tier 1 — State DB + recent messages (~10K tokens):
Tier 0 + sessions_history(limit=30, includeTools=false)

Tier 2 — Extended recovery (~15K tokens):
Tier 1 + read memory/daily-loop/tonight-recap.md

Tier 3 — Full boot (morning only, ~70-140K tokens):
Read yesterday + today conversation logs. Only when context is near-empty.

⚠️ ANTI-LOOP: Never reload more than needed or you'll trigger another compaction.

Each agent should query state DB at the start of every session/cron:

Scaling to Multiple Agents

For multiple agents, you have two options:

Option A: One cron extracts for multiple agents (recommended)

One Sonnet cron reads from multiple session keys and writes to each agent's own state.db. Cheaper, simpler.


Cron reads → Agent 1 session → writes to Agent 1 state.db
           → Agent 2 session → writes to Agent 2 state.db

Option B: Each agent self-writes (for less active agents)

Add the boot/end protocol to the agent's AGENTS.md. They write their own findings at the end of each cron run. No extra extraction cost, but depends on the agent remembering.

Best practice: Use Option A for agents with live conversations or high cron volume. Use Option B for agents that run 1-3 crons per day.

The Rules Engine — Scored Behavioral Memory

State extraction captures what happened. But some learnings need to persist forever — behavioral rules that should never be forgotten regardless of session, compaction, or context limits.

The problem: a flat rules file grows unbounded. A full DB dump gets too large for context. You need scored, decaying rules where only the most important ones load on boot.

Schema


CREATE TABLE rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule TEXT NOT NULL,                    -- The actual behavioral rule
    domain TEXT DEFAULT 'ALL',             -- Which domain it applies to
    source TEXT,                           -- rejection, learning, discovery
    source_id INTEGER,                     -- Original state entry ID
    score REAL DEFAULT 5.0,                -- 1-10 importance score
    violations INTEGER DEFAULT 0,          -- Times violated after creation
    last_reinforced TEXT,                   -- Last time user reinforced this
    last_relevant TEXT,                     -- Last time rule was applied
    created_at TEXT DEFAULT (datetime('now', 'localtime')),
    expires_at TEXT,                        -- NULL = never expires
    status TEXT DEFAULT 'active'
        CHECK(status IN ('active', 'dormant', 'retired', 'critical')),
    tags TEXT                              -- Comma-separated for filtering
);

-- Boot view: top 20 by score, only active/critical (~400 tokens even at 100+ rules)
CREATE VIEW v_boot_rules AS
SELECT id, rule, domain, score, status, tags
FROM rules 
WHERE status IN ('active', 'critical')
ORDER BY score DESC
LIMIT 20;

-- Domain-specific view (load when working in a specific domain)
CREATE VIEW v_domain_rules AS
SELECT id, rule, domain, score, tags
FROM rules
WHERE status IN ('active', 'critical')
ORDER BY domain, score DESC;

The Promotion Pipeline


Conversation → State Extractor (30 min) → state.db (rejections, learnings)
                                              ↓
                                    Nightly Rule Promotion (11 PM)
                                              ↓
                                        rules table (scored, tagged)
                                              ↓
                                    v_boot_rules (top 20, ~400 tokens)
                                              ↓
                                    Every boot loads ONLY what matters

Nightly Rule Promotion evaluates each day's rejections and learnings:

  • User said NO → score 9-10, status critical, auto-promote
  • Formatting/pipeline preference → score 7-8, status active
  • Operational learning → score 6-7, status active
  • One-time context → NOT promoted (stays in state.db only)

Score Decay

Rules that aren't reinforced naturally fade:

  • Every 30 days without reinforcement: score drops by 0.5
  • Score below 4 → auto-dormant
  • Score below 1 → auto-retired
  • Critical rules NEVER decay — only the user can retire them

Reinforcement

When the user repeats a preference or re-states a rule, the score goes UP:


UPDATE rules SET score = MIN(10, score + 1.0), 
    last_reinforced = datetime('now', 'localtime'),
    status = CASE WHEN score + 1.0 >= 9 THEN 'critical' ELSE status END
WHERE id = X;

Why This Scales

  • Boot cost is constant: v_boot_rules always returns max 20 rules (~400 tokens)
  • Even with 1,000+ rules: only the top 20 by score load on boot
  • Domain filtering: when working on trading, load only HODL rules
  • Natural memory: important things stay. Unimportant things fade. Just like human memory.

Boot Sequence


# Step 0: Load top rules (~400 tokens)
sqlite3 memory/state.db -header -column "SELECT * FROM v_boot_rules;"

# Domain-specific (when entering a domain):
sqlite3 memory/state.db -header -column \
    "SELECT rule, score FROM v_domain_rules WHERE domain = 'HODL';"

Anti-Patterns (What We Tried That Failed)

1. "Just write to a file after each conversation" — Agents forget. The memory problem prevents its own solution.

2. "Read the full conversation log after compaction" — 70K+ tokens causes an infinite compaction loop.

3. "Summarize the day at the end" — By the time "end of day" arrives, context is already lost.

4. "Use a TODO list" — Same as #1. Nobody maintains it.

The Breakthrough Insight

Separate the note-taker from the worker. The worker (main agent) can't reliably take notes because the same context loss that creates the problem prevents the solution. An independent process (the extraction cron) takes notes on a fixed schedule regardless of what the worker remembers.

This is the same principle as: don't ask the surgeon to take their own notes during surgery. Have a scribe.

What Makes Our Approach Different

Every other solution falls into one of two traps:

Trap 1: Relies on the agent to write memories.

MEMORY.md, ClawVault, Obsidian — they all require the agent to take notes. But the agent that needs memory is the same agent that forgets to create it. This is like asking someone with amnesia to remember to write in their journal. It works when they remember. They usually don't.

Trap 2: Relies on summarization (lossy compression).

LanceDB-pro, Claude-Mem, memory-mcp, Anthropic Compaction, Mem0 — they all use LLMs to summarize conversations into compressed memories. Summaries lose detail. "Jonathan rejected the Sonnet downgrade for Tom" becomes "discussed model options" after compression. The rejection — the most critical piece — is gone.

Our approach avoids both traps:

1. Automated extraction, not agent discipline. An independent cron (the "court stenographer") extracts state every 30 minutes. The worker agent doesn't need to remember to take notes. The note-taker is a separate process.

2. Structured data, not summaries. We don't summarize conversations into prose. We extract structured facts: decisions, tasks, rejections, agent states. Each one is a row in SQLite with a category, domain, status, and timestamp. You can query "show me all rejected suggestions" — try doing that with a summary.

3. Rejections are first-class citizens. No other system tracks what the user said NO to. This is arguably the most important memory an agent can have. Re-suggesting something the user rejected is the fastest way to destroy trust.

4. Tiered recovery prevents the infinite loop. Most systems either reload everything (expensive, risks compaction loop) or reload nothing (useless). Our tiered approach starts with a ~1K token DB query and only escalates if needed.

5. Zero infrastructure. SQLite. That's it. No vector database, no cloud service, no API keys, no embedding models. The state DB is a single file that works offline, can be version-controlled, and is human-readable with any SQLite client.

6. Watermark-based gap recovery. If the cron fails 3 times in a row, it doesn't lose data. It picks up from the last successful watermark and processes everything it missed. Every other system either loses the gap or re-processes everything.

The Research Backs This Up

  • 65% of enterprise AI failures in 2025 were from context drift/memory loss, not raw context limits (Zylos AI)
  • Factory's anchored iterative summarization (structured state document updated incrementally) outperforms full-reconstruction summarization — scoring 4.04 vs 3.74 accuracy. Our State DB IS this pattern, but with structured data instead of prose.
  • JetBrains research found that simple observation masking often outperforms LLM summarization while being cheaper — because summaries add cost and latency without reliable accuracy gains. Our structured extraction is closer to masking (preserve the signal, drop the noise) than summarization.
  • The "separate the note-taker from the worker" principle appears in Mem0's architecture (asynchronous summary generation module operating independently of the main pipeline) and memory-mcp's hook system. We implement it as a cron — simpler, more reliable, framework-agnostic.

Honest Limitations

  • 30-minute extraction gap. If compaction happens 29 minutes after the last extraction, you lose up to 29 minutes of state. The pre-compaction flush mitigates this but depends on agent compliance.
  • Extraction quality depends on the model. Sonnet is good but not perfect. Subtle rejections or implied decisions may be missed.
  • No semantic search. The State DB is structured queries only. For "what did we discuss about Shopify last week?" you still need vector search (QMD/memory_search). The State DB answers "what's active right now?" — a complementary role.
  • New system. Built March 8, 2026. Needs production hardening. The cron could fail, the prompts could extract poorly, edge cases will emerge.
  • Rule decay is approximate. The 30-day decay window is tunable but may need adjustment based on real-world observation of which rules fade too fast or too slow.

Part 2: The Capability Routing System

Added March 8, 2026 — same day as State DB, because the same problem applies to tools.

The Problem (Again)

The State DB solves context memory — what was decided, what's active, what was rejected. But there's a second category of knowledge agents lose: capabilities.

Our trading agent Tom has 12 scanner scripts, an IBKR integration, Unusual Whales API, Twitter monitoring, chart analysis tools — all installed, all working. He spent a week not using half of them because he forgot they existed. We had to manually tell him "you have this tool, USE IT."

Our coordinator Orion has an iPhone mirroring capability. Every time he uses it, he forgets that coordinates need to be divided by 2 for Retina displays. That lesson is buried in a conversation log from days ago.

Static CAPABILITIES.md files don't solve this because:

1. They're stale by definition — tools get added, break, change

2. They don't know what worked last time or what failed

3. They load everything at boot, wasting context on irrelevant capabilities

4. They don't track hard-earned lessons ("divide by 2 for Retina")

5. Agents don't maintain them

The Architecture

A SQLite database (capabilities.db) with 7 normalized tables:


┌─────────────────────────────────────────────────┐
│              capabilities.db                     │
│                                                  │
│  toolkits        — what exists                   │
│  agent_access    — who has what, at what tier     │
│  triggers        — FTS5 full-text keyword match   │
│  commands        — exact working commands          │
│  lessons         — hard lessons, per-agent         │
│  execution_log   — auto-captured success/failure   │
│  workflows       — multi-toolkit compositions      │
└─────────────────────────────────────────────────┘

Three Tiers

Not every capability loads every session. Like hard rules vs regular state:

1. Hard Capabilities — always loaded at boot. Tom always knows about IBKR. Muse always knows about fal.ai. These are the tools that define what the agent IS.

2. Task Capabilities — loaded on demand via FTS5 trigger matching. When Tom gets "scan for trades," the scanner toolkit loads. When he gets "execute a trade," the IBKR order toolkit loads. The agent gets exactly what's relevant, nothing more.

3. Reference Capabilities — searchable but not pre-loaded. Edge cases, old tools, rarely used features. Found via query when needed.

The same toolkit can be different tiers for different agents. IBKR is hard for Tom but reference for Orion.

Per-Agent Lessons

The key insight: lessons are first-class entities, not comments in a file.


CREATE TABLE lessons (
    id INTEGER PRIMARY KEY,
    toolkit_id INTEGER REFERENCES toolkits(id),
    agent TEXT,           -- per-agent: Tom's IBKR lessons ≠ Orion's
    severity TEXT,        -- 'critical', 'warning', 'info'
    lesson TEXT,          -- "Must set asyncio loop BEFORE importing ib_insync"
    supersedes INTEGER,   -- newer lesson replaces older one
    learned_at TEXT,
    times_relevant INTEGER DEFAULT 0
);

When Tom loads the IBKR toolkit, he gets Tom's lessons. When Orion loads it, he gets Orion's lessons. The Retina divide-by-2 lesson lives in Orion's iPhone mirroring toolkit and surfaces every time he touches that capability.

How It Stays Current

The biggest risk with any agent knowledge system: the agents won't maintain it. We solved this with four mechanisms at different timescales:

1. Real-time self-reporting — agents write critical lessons directly to capabilities.db the moment they learn them. Same pattern as state DB writes. Only for critical stuff — hard lessons, gotchas, auth breakthroughs.

2. Every 30 minutes — the capability extractor runs alongside the state extractor cron. Scans recent sessions for tool executions, successes, failures, lessons. Catches everything agents forgot to self-report.

3. Twice daily (10 PM + 6 AM) — the validator tests actual tool health. Runs test commands, checks dependencies (is IB Gateway running? Is the API key valid?). Flags unhealthy toolkits so agents know before they waste time trying.

4. On boot — agent queries capabilities DB for hard tier + task-matched capabilities via FTS5. One SQL call, relevant context only.

Dependency & Conflict Resolution

Toolkits declare their dependencies:


{
  "dependencies": [
    {"type": "process", "name": "IB Gateway", "check": "pgrep -f ibgateway"},
    {"type": "keychain", "name": "unusual-whales-api-key"}
  ],
  "conflicts": [
    {"type": "toolkit", "name": "ibkr-paper", "reason": "Same clientId"}
  ]
}

Before an agent uses a toolkit, dependencies are checked. Before two toolkits run simultaneously, conflicts are checked. No more "IBKR connection failed" after 5 minutes of setup because the gateway was down the whole time.

Workflow Composition

Real tasks aren't single tools — they're sequences. "Execute a trade" means:

1. Check market data (Yahoo Finance toolkit)

2. Analyze chart (chart-analyzer toolkit)

3. Check options flow (Unusual Whales toolkit)

4. Size the position (risk rules toolkit)

5. Place the order (IBKR toolkit)

6. Set stops (stop monitor toolkit)

7. Log the trade (trade journal toolkit)

The workflows table defines these compositions with ordered steps and prerequisites. An agent loading "trade execution" gets the full playbook, not just one tool.

Scale Analysis

At 1,000 toolkits, the context budget matters:

  • Hard tier: capped at 20 per agent (~4K tokens)
  • Task tier: top 5 FTS5 matches (~2K tokens)
  • Total boot context: ~7K tokens — well within budget

FTS5 queries return in <1ms on SQLite. The capability check adds negligible latency to agent boot.

Integration with State DB and Cognee

The three systems are complementary:

SystemAnswersData Type
State DB"What's happening? What was decided? What's rejected?"Structured decisions, rules, context
Capabilities DB"What can I do? How? What went wrong last time?"Tools, commands, lessons, health

Capability events feed Cognee too — every toolkit becomes a node in the knowledge graph. An agent can ask "what tools does Tom use for scanning?" and get a graph traversal answer, not just a keyword match.

Why This Matters

Without capability routing, agents are like employees who show up every day having forgotten they have a computer. They have the skills. They have the tools. They just don't know it until someone reminds them.

The State DB made agents remember what happened. The Capability Routing System makes them remember what they can do.

Part 3: The Flows DB — Procedural Memory

The Problem (Yet Again)

State DB tells agents what happened. Capabilities DB tells them what they can do. But neither tells them how to do it correctly.

An agent told to "deploy the blog" might:

  • FTP directly to production (skipping staging)
  • Overwrite existing content instead of adding to it
  • Skip QA entirely
  • Forget to git commit

Every time an agent does a task for the first time, it figures out the procedure from scratch. Every time it does the same task again after context loss, it figures it out again — sometimes differently, sometimes worse.

The Solution

Flows DB — a SQLite database of procedural recipes keyed by trigger phrases.


CREATE TABLE flows (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT UNIQUE NOT NULL,
  domain TEXT DEFAULT 'ALL',
  agent TEXT DEFAULT 'any',
  trigger_phrases TEXT,           -- comma-separated keywords
  needs_approval BOOLEAN DEFAULT 0,
  steps TEXT NOT NULL,            -- JSON array of ordered steps
  status TEXT DEFAULT 'active',
  use_count INTEGER DEFAULT 0,
  last_used TEXT,
  created_at TEXT DEFAULT (datetime('now','localtime'))
);

How It Works

Before any action, an agent runs flow-check.sh "what I'm about to do":


$ bash tools/flow-check.sh "deploy to ienable"

=== FLOW: deploy-to-ienable ===
  Step 1: BACKUP: cp file to workspace/backups/ with timestamp
  Step 2: READ: cat the file first. Understand what's there.
  Step 3: EDIT: Surgical additions ONLY. Never rebuild.
  Step 4: BUILD: npm run build in ienable-site
  Step 5: GIT: git add -A && git commit && git push
  Step 6: HAND TO QA: Spawn qa agent with URLs to check.
  Step 7: QA MUST PASS before reporting to Jonathan.

The flow is the procedure. The agent doesn't need to remember how to deploy — the flow remembers for it.

Key Design Decisions

Trigger phrases, not exact matches. A flow for "seo optimize post" matches queries containing "seo," "optimize," or "post." This means agents find flows even when they phrase things differently.

Needs-approval flag. Some flows (deploying to production, placing live trades, publishing strategic content) require human sign-off. The flow itself enforces the gate — the agent can't skip it because the procedure says "WAIT for approval."

Use tracking. Every time a flow is used, use_count increments and last_used updates. Flows that are never used are either poorly triggered or unnecessary.

Agent-specific flows. A flow for agent=hodl won't fire for the content agent. Domain separation is maintained.

Part 4: The Pre-Action Gate — Mandatory Context Loading

The Problem (The Deepest One)

All three systems above — State DB, Capabilities DB, Flows DB — share one fatal flaw: they only work if the agent checks them.

An agent with perfect memory that it never consults is an agent with no memory at all.

In practice, agents rush to execute. A directive comes in ("fix Tom's crons") and the agent's next action is a tool call, not a context load. The knowledge is there. The rejection history is there. The flows are there. But the agent bypasses all of it because executing feels like progress.

This is the same reason humans skip checklists. It's not ignorance — it's urgency bias.

Part 5: The Post-Action Gate — Grading, Consolidation, and Compaction

The Missing Half

Parts 3 and 4 solved the intake problem: agents load context before acting. But nothing happened after the action completed. No grading. No flow updates. No lesson consolidation. The system accumulated endlessly — 449 lessons in one agent's database, 240 active state entries in another, rules that were never reinforced but never retired.

The Pre-Action Gate was the intake valve. The Post-Action Gate is the exhaust valve. Without both, the system fills up with noise.

post-action.sh — Three Gates After Every Task

Gate 1: Grade. Pass or fail. Failures get logged to state DB for investigation. Simple binary — did the task succeed?

Gate 2: Update flow. If a flow was used, increment its use counter. If no flow was used, create a draft placeholder so the next agent doesn't start from scratch. This is how the flow library grows organically — every unflowed task flags itself for documentation.

Gate 3: Log state. One consolidated entry, not five. If a similar state entry already exists, update it instead of creating a duplicate. If a lesson is provided, check for an existing similar rule and reinforce it instead of adding a new one. New lessons start at score 5 — they must be reinforced to survive nightly compaction.

The Consolidation Principle

The key design decision: reinforce, don't duplicate.

When an agent learns "always read before deleting," the system checks if a similar rule already exists. If it does, the existing rule's score increases by 0.5 (up to a maximum of 10). If it doesn't, a new rule is created at score 5.

This means:

  • Lessons that keep getting validated rise to the top
  • Lessons that were situational fade over time
  • The database never grows unboundedly
  • Important rules are self-evident by their score

What Triggers a Post-Action?

Three categories:

1. Cron completions — every scheduled task that announces back

2. Sub-agent completions — every spawned task that returns a result

3. Direct human directives — when an agent executes something the user asked for

The first two are automatable (they already flow through defined completion paths). The third requires the agent to self-identify "I just finished something" — which is fragile by design. The long-term fix is platform-level hooks that fire automatically on session end, removing the agent from the responsibility entirely.

Part 6: Nightly Compaction — The System Gets Sharper, Not Bigger

compact-rules.sh — The Nightly Janitor

Runs once per day on every agent database. Four phases:

Phase 1: Decay

Rules not reinforced in 7 days lose 0.5 points per cycle. Score-10 rules (critical, hard-won) are exempt from decay. This creates natural selection pressure — only rules that keep proving themselves survive.

Phase 2: Retire

Rules that decay below score 1 are deleted. They had their chance. If they were important, they would have been reinforced by the post-action gate catching the same lesson again.

Phase 3: Merge

Near-duplicate rules (same first 40 characters) are consolidated. The higher-scored version survives and gets a 0.5 boost. This prevents the "10 slightly different versions of the same lesson" problem.

Phase 4: Archive State

Completed state entries older than 3 days move to archived status. The active state view stays lean — only current work and recent decisions. Historical context is preserved but doesn't clutter the boot sequence.

The Math

Without compaction: a system generating 5 lessons/day accumulates 1,825/year. With compaction: the steady state is roughly 60-100 rules, where only the most reinforced and relevant survive. The database is bounded by selection pressure, not by time.

The Compaction Algorithm (Specified)


FOR EACH agent database:
  1. SELECT rules WHERE score < 10 AND last_reinforced > 7 days ago
     → Decrease score by 0.5
  
  2. DELETE rules WHERE score ≤ 1
     → These failed to prove value
  
  3. SELECT pairs WHERE same domain AND first 40 chars match
     → Keep higher score, delete lower, boost survivor +0.5
  
  4. UPDATE state SET status='archived' WHERE completed AND older than 3 days

No fuzzy matching. No LLM-in-the-loop. Pure SQL operations that run in milliseconds. The compaction cron uses haiku (cheapest model) because it's just running shell commands.

The Insight Behind Everything

Every system we built addresses the same root cause: AI agents are not responsible enough to remember to remember.

  • State DB automates state extraction (the agent doesn't write its own notes)
  • Rules Engine promotes lessons automatically (the agent doesn't self-improve)
  • Capabilities DB is maintained by a separate cron (the agent doesn't catalog itself)
  • Flows DB encodes procedures once (the agent doesn't re-derive steps)
  • The Pre-Action Gate loads all of it automatically (the agent doesn't choose to check)
  • The Post-Action Gate grades and consolidates (the agent doesn't self-assess)
  • Nightly Compaction prunes and sharpens (the agent doesn't curate its own knowledge)

At every layer, the design principle is the same: remove the agent from the responsibility of maintaining its own memory. External systems observe, extract, store, and serve. The agent just shows up and gets context.

That's the entire philosophy. Don't trust the agent to remember. Build systems that remember for it. And don't trust the system to stay lean — build compaction that keeps it sharp.

Part 7: The Meta-Lesson — Why Lessons Don't Stick

The Evidence

On February 19, 2026, Orion logged this lesson:

> "ALWAYS check CAPABILITIES.md + memory_search() before installing new tools."

On February 25, 2026:

> "Always memory_search before asking or researching."

On March 10, 2026 — 19 days later — Orion told Jonathan that the Unusual Whales API key was missing (it was in the keychain), that GA4 needed a service account (it had been at ~/.openclaw/credentials/ since February 8), and that several capabilities needed to be built (they already existed in agent workspaces).

The lessons were recorded. They were never made infrastructure. A rule in a database that says "check first" is behavioral compliance. The agent has to choose to read the rule, choose to follow it, and execute the check correctly every time. That failed 100% of the time over 19 days.

The Fix

The fix was not another rule. The fix was adding an Existing Tools Check directly into pre-action.sh (Gate 6) that automatically:

1. Searches every wrapper script by keyword

2. Searches every agent's tools directory

3. Checks the macOS Keychain for matching credentials

4. Displays everything found before the agent starts working

The agent doesn't choose to check. The gate checks for it. If uw-api.sh exists and the keychain has unusual-whales-api-key, the gate shows both — before the agent even begins.

The Pattern

This is the same pattern at every layer:

  • Rules decay. A score-10 rule was logged on Feb 19. By Mar 10, the behavior hadn't changed.
  • Reflections feel like action. Writing "I'll check next time" feels productive but changes nothing.
  • Infrastructure persists. Adding the check to pre-action.sh means every agent, every session, every task sees existing tools automatically.

Gate 4: The Enforcer

Post-action now has a fourth gate that asks: "Did you change a script, flow, or schema? Or did you just add a rule?"

If the answer is "just a rule," the lesson is not permanent. The gate challenges the agent to convert the lesson into infrastructure. This gate exists because:

1. February 19: Rule added. No infrastructure change.

2. February 25: Rule reinforced. No infrastructure change.

3. March 10: Same failure. Jonathan had to push three times to get the infrastructure change.

The lesson about checking capabilities was itself an example of a lesson that didn't stick because it wasn't infrastructure. The recursion is intentional.

The Honest Count

How many lessons were lost between February 1 and March 10? Unknown. There is no audit trail for "agent said it learned something but only added a rule." The only evidence is when the same failure recurs and the previous lesson is found in the logs.

The system now has tools to prevent this:

  • pre-action.sh Gate 6: Existing tools check (prevents rebuilding)
  • post-action.sh Gate 4: Infrastructure or behavioral? (prevents false learning)
  • compact-rules.sh: Decays rules that aren't reinforced (removes noise)
  • infra-audit.sh: Shows what's infrastructure vs behavioral (makes the gap visible)

None of these existed before March 10. All of them exist because the same lesson failed to stick four times.

Part 8: The Heartbeat — Why Context Recovery Needs a Pulse (March 12, 2026)

The Problem That Parts 1-7 Didn't Solve

Every system above is pull-based. State DB, Flows DB, Capabilities DB, Pre-Action Gate — they all require the agent to actively query them. After compaction, the agent has to know it needs context and go get it.

But here's the failure mode we discovered on March 11-12: the agent compacts mid-task, the heartbeat is disabled, and nothing wakes it up. The conversation goes silent. The agent doesn't stall because it's confused — it stalls because nothing triggers the next turn. The user has to manually ping to restart work.

We tried solving this with a cron job that fired every 15 minutes. That spawned a new isolated session each time — a fresh agent with no context, no conversation history, and no way to continue the work. Dozens of these ghost sessions burned tokens doing nothing useful.

The Breakthrough: Push-Based Context Recovery

OpenClaw's heartbeat mechanism runs periodic turns in the main session — not isolated spawns. Every 5 minutes, a heartbeat fires into the existing conversation, with full session context preserved. This is the pulse that keeps the agent alive.

Combined with a smart context loader (context-loader.sh), each heartbeat:

1. Loads the last 100 messages OR last 12 hours of conversation (whichever is more)

2. Loads active state from state.db

3. Checks for active task checkpoints

4. If mid-task: continues working immediately

5. If idle: replies HEARTBEAT_OK (suppressed, costs ~$0.15-0.25)

Why This Changes Everything

The Pre-Action Gate (Part 4) was the intake valve — it loaded context before actions. But it only fired when the agent was about to DO something. After compaction, the agent doesn't know it needs to do anything. It's waiting for input that never comes.

The heartbeat is the clock signal. It fires regardless of what the agent knows or doesn't know. It's the external trigger that says "wake up, check your state, continue if needed." The Pre-Action Gate handles WHAT to load. The heartbeat handles WHEN to check.

The Missing Layer: Ambient Memory (lancedb-pro)

Parts 1-7 are all structured, pull-based systems. You query state.db. You run pre-action.sh. You check capabilities.db. They require the agent to take action.

lancedb-pro's auto-recall is ambient and push-based. Every turn, before the agent even sees the prompt, the plugin:

1. Takes the incoming message

2. Runs hybrid retrieval (vector + BM25 keyword search)

3. Cross-encoder reranks the results

4. Injects the top-5 most relevant memories as context

The agent doesn't choose to search its memory. Memory comes to it. After compaction, the very first message triggers auto-recall, which surfaces the most relevant recent context. Combined with the heartbeat firing every 5 minutes, this means:

  • Within 5 minutes of compaction: the heartbeat fires, auto-recall injects relevant memories, and the agent has enough context to continue
  • No scroll-up needed for most cases — the ambient layer handles it
  • Deep context available on demand via context-loader.sh, state.db, and conversation files

Part 9: The Dispatch System — Capability → Agent → Flow Routing (March 12, 2026)

The Problem That Parts 1-8 Still Don't Solve

Every system above answers a different question: State DB knows what happened. Capabilities DB knows what tools exist. Flows DB knows how to execute procedures. Rules Engine knows what not to do. The heartbeat keeps the agent alive. lancedb-pro surfaces relevant memories.

But none of them answer the simplest question a coordinator agent faces 50 times a day: "Who does this and how?"

When the user says "make a logo," the coordinator has to mentally reconstruct three things:

1. What tool? (fal.ai FLUX Pro — buried in SOUL.md or capabilities.db)

2. Which agent? (Muse — known from AGENTS.md but not linked to the capability)

3. What's the proven command? (exact curl syntax — last used 3 sessions ago, since lost to compaction)

These three facts exist in three separate systems. The coordinator has to JOIN them in its head every time. After compaction, one or more links break and the coordinator either:

  • Spawns a disposable sub-agent instead of routing to the persistent agent (wrong WHO)
  • Rediscovers the tool from scratch (wrong HOW)
  • Asks the user (wastes time)

Evidence: On March 12, 2026, the coordinator spawned 6 throwaway sub-agents for frontend tasks that should have gone to Alex (fullstack-dev) — a persistent Opus agent with his own memory, rules DB, and deep context about the codebase. A score-10 rule saying "ALWAYS spawn agents with agentId" was in the rules DB. The coordinator loaded it on boot. And still spawned anonymous sub-agents every time. The rule existed. The behavior didn't change.

Why Rules Don't Fix This

Part 7 ("The Meta-Lesson") already identified this pattern: lessons that aren't infrastructure don't stick. A rule saying "route frontend tasks to Alex" is behavioral compliance. The agent has to remember the rule, identify the task as "frontend," recall Alex's session key, and use sessions_send instead of sessions_spawn. Four decision points, each one a chance to default to the easier path.

The easier path is always sessions_spawn — it's what the coordinator uses for backtests (disposable tasks), and the muscle memory carries over to persistent agent work.

The Solution: dispatch.db

A single SQLite table that eliminates the decision chain entirely:


CREATE TABLE dispatch (
    id INTEGER PRIMARY KEY,
    task_keywords TEXT NOT NULL,        -- "logo,image,graphic,design"
    task_domain TEXT NOT NULL,           -- "MAXWOOD", "IENABLE", "HODL", "PERSONAL", "ALL"
    agent_id TEXT NOT NULL,             -- "muse", "fullstack-dev", "hodl"
    agent_session TEXT NOT NULL,        -- "agent:muse:main"
    capability TEXT NOT NULL,           -- "image-generation"
    tool TEXT,                          -- "fal.ai FLUX Pro"
    proven_command TEXT,                -- Exact curl/command that works
    flow_name TEXT,                     -- Reference to flows.db
    flow_location TEXT,                 -- Path to .lobster or procedure file
    last_used DATETIME,
    use_count INTEGER DEFAULT 0,
    confidence REAL DEFAULT 1.0,
    created_at DATETIME DEFAULT (datetime('now','localtime'))
);

One query returns everything: agent + session key + tool + command + flow. No mental JOIN. No rediscovery. No decision chain.

When Teams ships, dispatch.db becomes the seed data for team configuration — the default routing table that Teams can override dynamically.

Part 10: North Star Missions — Agents That Don't Wait for Instructions (March 13, 2026)

Parts 1-9 solved memory loss. The agents could remember. But they still sat idle waiting to be told what to do.

The Problem

An agent with perfect memory and zero initiative is an expensive note-taker. Every morning, the coordinator (Orion) would wake up, read state.db, and manually assign tasks to each agent. If Orion didn't assign work, agents did nothing. If Orion was busy, agents were idle. 9 agents × 8 idle hours = 72 wasted agent-hours per night.

The Pattern

Each agent gets a North Star Mission baked directly into their AGENTS.md — a permanent directive that survives every restart, compaction, and context loss:


## NORTH STAR MISSION (March 13, 2026)

iEnable must rank #1 on Google AND #1 in LLM citations 
for "AI enablement" and every related keyword.

### Self-Directed Work
You do NOT need to wait for Orion to give you tasks. Every night:
1. Check Google Search Console for new data
2. Identify gaps: keywords with impressions but no clicks
3. Audit existing posts: are titles, metas, H1s optimized?
4. Create NEW content targeting unowned keywords

The North Star isn't a task list — it's a permission slip. It tells the agent: "You know your domain. If you see something that needs doing, do it."

The Result

Apollo (content agent) immediately self-directed: found 12 dead posts, fixed 33 duplicate H1 tags, audited internal links (8 orphans, 14 weak, 48 healthy), validated the sitemap — all without a single instruction from the coordinator. Not because he was told to, but because his North Star said "rank #1" and he looked at the data.

The Rule

Every agent needs three things to self-direct:

1. A North Star in AGENTS.md (what they're trying to achieve)

2. Loaded capabilities on boot (what they can do)

3. Data access (what the current state is)

Remove any one of these and the agent reverts to waiting for instructions.

Part 15: The Universal Agent Boot — One Script, Full Context (March 13, 2026)

The Problem

Each agent's cron had ad-hoc sqlite queries pasted into the prompt — different queries, different databases, inconsistent coverage. Some agents loaded rules but not capabilities. Some loaded lessons but not rejections. No agent loaded all five context layers.

The Solution: agent-boot.sh

A single script that loads everything an agent needs to operate:


bash tools/agent-boot.sh <agent_id>

What it loads (in order):

1. Rules — Top 15 by score for the agent's domain + ALL. Critical rules (10) appear first.

2. Capabilities — Hard-tier tools for this agent, with health status. The agent knows what's working and what's broken before attempting anything.

3. Toolkit Lessons — Critical lessons from capabilities.db linked to tools this agent uses. "IBKR requires reauth every 24h" loads for Tom, not for Apollo.

4. Agent Lessons — Top 10 from the agent's own learning.db. Hard-won, domain-specific knowledge.

5. Flows — All active flows for this agent, backtest-extracted flows first. The agent sees its proven procedures before improvising.

6. Rejections — What the human said NO to. Auto-loaded, never re-suggested.

Capability Access Tiers

Every agent has explicit capability mappings:

TierMeaningWhen Loaded
hardAlways loaded at bootEvery session, every cron
taskLoaded when relevantWhen the task matches the capability
referenceQueryable but not injectedOn-demand only

Access mappings ensure each agent's context window is spent on relevant capabilities, not cluttered with tools they'll never use. Tom gets trading tools. Apollo gets SEO tools. Neither wastes tokens on the other's stack.

Integration

Every agent's HEARTBEAT.md now starts with:


bash tools/agent-boot.sh <agent_id>

bash tools/post-action.sh <agent_id> <pass|fail> "task" "flow" "lesson"

The boot script is the single entry point. One script to rule them all.
