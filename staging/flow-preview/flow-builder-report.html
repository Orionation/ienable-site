1. Design Philosophy

Core Principle: "Pipes With Quality Control"

Every workflow builder in the market optimizes for speed of execution. iEnable optimizes for quality of output. This is not a constraint — it's a feature. The fastest pipeline that ships garbage is worthless. A pipeline that takes 10 extra minutes but ships great work is priceless.

Three Axioms

  1. The generator never grades its own homework. Every AI output should be independently evaluated.
  2. Approval is not an exception — it's the default. Gates start at Human, earn their way to Auto.
  3. Rejection is the most valuable data in the system. Every "no" makes the next attempt better.

Design Influences

  • Make.com: Visual builder UX, real-time execution visualization, subscenarios
  • n8n: Data pinning, execution-based mental model, self-hosting option
  • Relay.app: HITL as first-class primitive (we extend this concept dramatically)
  • Retool: Platform approach (apps + workflows + data), User Task blocks
  • Dify: Plugin SDK, parallel execution, visual debugging
  • Zapier Canvas: NL → flow generation, ROI tracking

A quality checkpoint. The flow pauses until the gate passes. See §4 for full spec.

Gate Mode Who Evaluates Example
Human Person reviews + approves/rejects Marketing director reviews ad copy
AI QA Independent AI agent evaluates QA agent scores brand compliance
Auto-pass Automatic (trust earned) Proven flow step, skip review
Hybrid AI pre-screens, human decides edge cases AI flags issues, human makes call

🔴 Branch

Conditional logic. Split the flow based on data.

Branch Type Description Example
If/Else Binary condition "If product price > $500..."
Switch Multiple conditions Route by product category
A/B Split Random percentage split 50/50 test two ad variants
Filter Pass/block based on criteria "Only products with images"

🟣 Loop

Repeat over a set of items.

Loop Type Description Example
For Each Iterate over array Process each product in catalog
While Repeat until condition Retry generation until QA passes
Batch Process in chunks Generate 10 ads at a time
Retry Automatic retry on failure Re-attempt API call 3x

🟤 Brief

A structured input document that precedes creative work. See §7.

⚪ QA

Independent quality evaluation. See §6.

🔷 Merge

Combine parallel branches back together.

Merge Type Description Example
Wait All Proceed when all branches complete Wait for all platform formats
Wait Any Proceed on first completion Use fastest image generation
Aggregate Combine outputs from all branches Collect all product descriptions

🔶 Subscenario

4. Approval Gates — The Killer Feature

Gate Configuration

Every gate has these properties:

gate:
  name: "Creative Brief Approval"
  mode: human | ai_qa | auto_pass | hybrid

  # Human mode config
  human:
    assignee: user | team | role | anyone_with_link
    notification_channels: [email, slack, push, sms]
    timeout: 48h
    escalation:
      - after: 24h → notify_manager
      - after: 36h → reassign_to: backup_reviewer
      - after: 48h → auto_action: reject_with_reason("timeout")
    delegation: true  # assignee can delegate to another user

  # AI QA mode config
  ai_qa:
    evaluator_agent: "brand_compliance_checker"
    criteria:
      - brand_voice_score: >= 7
      - factual_accuracy: true
      - platform_compliance: true
    pass_threshold: all_criteria_met
    on_fail: reject_with_structured_feedback

  # Auto-pass config
  auto_pass:
    conditions:
      - trust_score: >= 0.9  # earned over time
      - content_type: in [text_only, minor_edit]
      - risk_level: low
    audit_log: true  # still logs even when auto-passing

  # Hybrid config
  hybrid:
    ai_pre_screen: "brand_compliance_checker"
    if_ai_passes: auto_pass
    if_ai_flags: route_to_human
    if_ai_rejects: reject_immediately

  # Universal gate properties
  feedback:
    on_reject:
      require_reason: true
      reason_taxonomy: [off_brand, factually_wrong, wrong_tone, 
                        wrong_format, wrong_audience, low_quality,
                        missing_elements, other]
      free_text: optional
      reference_examples: optional  # reviewer can attach "do it like this"
    on_approve:
      score: optional  # 1-10 quality rating
      notes: optional  # "Great but next time try X"

  # Learning integration  
  learning:
    feed_rejection_to: generator_agent  # who created the content
    update_brand_context: true
    log_to_learning_db: true
    adjustment_weight: 0.1  # how much this feedback shifts behavior

Gate Evolution: The Trust Ladder

This is the concept no competitor has. Gates evolve based on track record:

STAGE 1: New Flow / New Agent
├── All gates: HUMAN
├── Every output reviewed by a person
└── Building the training dataset

STAGE 2: Patterns Emerge (after ~50 approvals)
├── Low-risk gates: HYBRID (AI pre-screens, human decides edge cases)
├── High-risk gates: HUMAN
└── AI QA agent learning from human decisions

STAGE 3: Trust Building (after ~200 approvals with >90% first-pass rate)
├── Low-risk gates: AUTO-PASS (with audit logging)
├── Medium-risk gates: AI QA
├── High-risk gates: HUMAN
└── System suggests gate promotions: "This gate has 95% approval rate — promote to AI QA?"

STAGE 4: Mature Flow (after ~500+ approvals)
├── Most gates: AUTO-PASS
├── Novel/unusual content: AI QA
├── High-stakes content: HUMAN
└── Anomaly detection: Auto-escalate when output looks unusual

The key insight: The customer controls the pace. They can override any promotion suggestion. They can demote gates back to Human at any time. Trust is earned, not assumed.

Gate Notification System

When gate is waiting for human:
  1. Instant notification via configured channels (Slack, email, push, SMS)
  2. Deep link directly to approval UI (mobile-friendly)
  3. Rich preview of content being reviewed (not just "you have an approval")
  4. One-tap approve with optional feedback
  5. Structured reject with required reason

Escalation chain:
  - T+0: Assigned reviewer notified
  - T+X (configurable): Reminder sent
  - T+Y: Manager/backup notified
  - T+Z: Auto-action (reject, reassign, or escalate)

Smart scheduling:
  - Respect reviewer's working hours
  - Batch non-urgent approvals into daily digest
  - Flag urgent approvals with distinct notification style

5. AI Agent Steps

Agent as a Flow Node

An Agent step is a special Action node that executes an AI agent with persistent memory and brand context.

agent_step:
  name: "Write Product Description"
  agent_id: "product_copywriter_v3"

  # Agent capabilities
  model: claude-sonnet-4  # or customer's preferred model
  temperature: 0.7

  # Brand context (loaded automatically)
  brand_context:
    brand_bible: "plankandbeam_brand_bible_v2"
    tone_guide: "warm, approachable, parent-focused"
    avoid: ["cheap", "discount", "basic"]
    examples: ["top_performing_descriptions.json"]

  # Memory
  memory:
    short_term: conversation_context  # within this flow run
    long_term: learning_db  # accumulated wisdom from all runs
    knowledge_graph: cognee_namespace  # brand knowledge graph

  # Input mapping
  inputs:
    product_data: "{{trigger.product}}"
    target_platform: "shopify"
    word_count: 150

  # Output schema
  outputs:
    description: string
    meta_title: string
    meta_description: string
    keywords: string[]

  # Learning integration
  learning:
    log_output: true
    track_downstream_approval: true  # did the gate after this pass?
    feedback_integration: "adjust_next_run"  # use rejection feedback

6. QA Steps

The Generator ≠ Grader Principle

The most important architectural decision in iEnable: the agent that creates content is NEVER the agent that evaluates it. This mirrors how every good creative organization works — the writer doesn't edit their own copy, the designer doesn't QA their own designs.

QA Node Configuration

qa_step:
  name: "Brand Compliance Check"
  evaluator_agent: "brand_qa_agent_v2"  # DIFFERENT from generator

  # What's being evaluated
  input: "{{product_copywriter.output}}"

  # Evaluation criteria (rubric)
  rubric:
    - criterion: "Brand Voice"
      description: "Does the copy match our warm, approachable tone?"
      weight: 0.3
      scoring: 1-10
      auto_fail_below: 4

    - criterion: "Factual Accuracy"
      description: "Are all product specs correct?"
      weight: 0.25
      scoring: pass/fail

    - criterion: "Platform Compliance"
      description: "Meets character limits, formatting rules for target platform?"
      weight: 0.2
      scoring: pass/fail

    - criterion: "SEO Quality"
      description: "Includes target keywords naturally?"
      weight: 0.15
      scoring: 1-10
      auto_fail_below: 5

    - criterion: "Differentiation"
      description: "Stands out from competitor copy?"
      weight: 0.1
      scoring: 1-10

  # Pass/fail logic
  passing:
    min_weighted_score: 7.0
    all_pass_fail_must_pass: true
    no_auto_fails: true

  # Output
  outputs:
    overall_score: number
    per_criterion_scores: object
    pass: boolean
    feedback: string  # detailed feedback for generator
    suggested_improvements: string[]

  # On fail
  on_fail:
    action: route_to_gate  # human reviews the QA failure
    # OR action: loop_back_to_generator  # auto-retry with feedback
    max_retries: 3
    include_feedback: true  # QA feedback sent to generator

QA Agent Independence

QA agents are: - Separate model instances — they don't share context with the generator - Separately prompted — their system prompt is about evaluation, not creation - Separately configured — different temperature (lower for QA), different model potentially - Separately learning — they learn what "good" looks like from human-approved outputs - Auditable — every evaluation is logged with full reasoning

QA → Gate Integration

The QA step feeds into a Gate:

Agent (Generate) → QA (Evaluate) → Gate (Decide)

If QA passes → Gate auto-passes (in AI QA mode)
If QA fails → Gate shows human the output + QA feedback
Human can:
  - Agree with QA → Reject (feedback to generator)
  - Override QA → Approve anyway (QA agent learns it was too strict)
  - Partially agree → Edit and approve (system learns the edit pattern)

This three-step pattern (Generate → QA → Gate) is iEnable's core content production primitive.

Learning Database Schema (Simplified)

-- Every output ever generated
CREATE TABLE flow_outputs (
    id UUID PRIMARY KEY,
    flow_id UUID,
    run_id UUID,
    agent_id UUID,
    node_id UUID,
    output_type TEXT,           -- text, image, video, etc.

CREATE TABLE gate_decisions (
    id UUID PRIMARY KEY,
    flow_id UUID,
    run_id UUID,
    gate_id UUID,

CREATE TABLE lessons (
    id UUID PRIMARY KEY,
    agent_id UUID,             -- which agent this applies to

    lesson TEXT,               -- the actual lesson
    source_decisions UUID[],   -- which decisions spawned this

    active BOOLEAN DEFAULT true,
    created_at TIMESTAMP
);

CREATE TABLE agent_performance (
    id UUID PRIMARY KEY,
    agent_id UUID,
    period DATE,
    total_outputs INTEGER,
    first_pass_approval_rate NUMERIC,
    avg_quality_score NUMERIC,
    common_rejection_reasons JSONB,
    gate_promotion_eligible BOOLEAN,
    created_at TIMESTAMP
);

-- Trust scores for gate evolution
CREATE TABLE gate_trust (
    id UUID PRIMARY KEY,
    gate_id UUID,
    current_mode TEXT,         -- human, ai_qa, auto_pass
    approval_rate_30d NUMERIC,
    total_decisions INTEGER,
    last_rejection TIMESTAMP,
    promotion_eligible BOOLEAN,
    promotion_suggested_at TIMESTAMP,
    promoted_at TIMESTAMP,
    demoted_at TIMESTAMP
);

Learning Loop Mechanics

  1. Immediate: Rejection feedback is structured (taxonomy + detail) and stored
  2. Per-Run: If a loop/retry step exists, rejection feedback is injected into the next attempt's prompt
  3. Per-Agent: After every N decisions, lessons are extracted and added to the agent's long-term memory
  4. Per-Flow: Monthly, gate trust scores are recalculated. Promotion suggestions surface in the Approval Queue
  5. Per-Organization: Cross-flow patterns emerge. If "off_brand" is the #1 rejection reason across all flows, the brand bible may need updating — system alerts the admin

Execution Engine

Flow Run Lifecycle:

1. TRIGGER fires → creates a Run instance
2. Engine resolves first node(s) → executes
3. For each node:
   a. Resolve inputs (data mapping from upstream nodes)
   b. Execute (API call, agent inference, etc.)
   c. Store output in run context
   d. Determine next node(s) via edges
4. When a GATE is reached:
   a. Pause execution at this branch
   b. Create approval request
   c. Notify assignee(s)
   d. Wait for decision (async — engine continues other branches)
   e. On decision: resume or loop back
5. When all branches complete → Run finishes
6. Post-run: learning loop processes all gate decisions

Key architectural choices: - Event-driven, not polling-based — gates use webhooks/websockets, never polling - Parallel branch execution — branches run concurrently, Merge nodes sync them - Durable execution — runs survive server restarts (state persisted to DB) - Cost tracking — every API call's cost is recorded and attributed to the run - Audit log — every action, decision, and data transformation is logged

1. Design Philosophy

Core Principle: "Pipes With Quality Control"

Every workflow builder in the market optimizes for speed of execution. iEnable optimizes for quality of output. This is not a constraint — it's a feature. The fastest pipeline that ships garbage is worthless. A pipeline that takes 10 extra minutes but ships great work is priceless.

Three Axioms

  1. The generator never grades its own homework. Every AI output should be independently evaluated.
  2. Approval is not an exception — it's the default. Gates start at Human, earn their way to Auto.
  3. Rejection is the most valuable data in the system. Every "no" makes the next attempt better.

Design Influences

  • Make.com: Visual builder UX, real-time execution visualization, subscenarios
  • n8n: Data pinning, execution-based mental model, self-hosting option
  • Relay.app: HITL as first-class primitive (we extend this concept dramatically)
  • Retool: Platform approach (apps + workflows + data), User Task blocks
  • Dify: Plugin SDK, parallel execution, visual debugging
  • Zapier Canvas: NL → flow generation, ROI tracking

4. Approval Gates — The Killer Feature

Gate Configuration

Every gate has these properties:

gate:
  name: "Creative Brief Approval"
  mode: human | ai_qa | auto_pass | hybrid

  # Human mode config
  human:
    assignee: user | team | role | anyone_with_link
    notification_channels: [email, slack, push, sms]
    timeout: 48h
    escalation:
      - after: 24h → notify_manager
      - after: 36h → reassign_to: backup_reviewer
      - after: 48h → auto_action: reject_with_reason("timeout")
    delegation: true  # assignee can delegate to another user

  # AI QA mode config
  ai_qa:
    evaluator_agent: "brand_compliance_checker"
    criteria:
      - brand_voice_score: >= 7
      - factual_accuracy: true
      - platform_compliance: true
    pass_threshold: all_criteria_met
    on_fail: reject_with_structured_feedback

  # Auto-pass config
  auto_pass:
    conditions:
      - trust_score: >= 0.9  # earned over time
      - content_type: in [text_only, minor_edit]
      - risk_level: low
    audit_log: true  # still logs even when auto-passing

  # Hybrid config
  hybrid:
    ai_pre_screen: "brand_compliance_checker"
    if_ai_passes: auto_pass
    if_ai_flags: route_to_human
    if_ai_rejects: reject_immediately

  # Universal gate properties
  feedback:
    on_reject:
      require_reason: true
      reason_taxonomy: [off_brand, factually_wrong, wrong_tone, 
                        wrong_format, wrong_audience, low_quality,
                        missing_elements, other]
      free_text: optional
      reference_examples: optional  # reviewer can attach "do it like this"
    on_approve:
      score: optional  # 1-10 quality rating
      notes: optional  # "Great but next time try X"

  # Learning integration  
  learning:
    feed_rejection_to: generator_agent  # who created the content
    update_brand_context: true
    log_to_learning_db: true
    adjustment_weight: 0.1  # how much this feedback shifts behavior

Gate Evolution: The Trust Ladder

This is the concept no competitor has. Gates evolve based on track record:

STAGE 1: New Flow / New Agent
├── All gates: HUMAN
├── Every output reviewed by a person
└── Building the training dataset

STAGE 2: Patterns Emerge (after ~50 approvals)
├── Low-risk gates: HYBRID (AI pre-screens, human decides edge cases)
├── High-risk gates: HUMAN
└── AI QA agent learning from human decisions

STAGE 3: Trust Building (after ~200 approvals with >90% first-pass rate)
├── Low-risk gates: AUTO-PASS (with audit logging)
├── Medium-risk gates: AI QA
├── High-risk gates: HUMAN
└── System suggests gate promotions: "This gate has 95% approval rate — promote to AI QA?"

STAGE 4: Mature Flow (after ~500+ approvals)
├── Most gates: AUTO-PASS
├── Novel/unusual content: AI QA
├── High-stakes content: HUMAN
└── Anomaly detection: Auto-escalate when output looks unusual

The key insight: The customer controls the pace. They can override any promotion suggestion. They can demote gates back to Human at any time. Trust is earned, not assumed.

Gate Notification System

When gate is waiting for human:
  1. Instant notification via configured channels (Slack, email, push, SMS)
  2. Deep link directly to approval UI (mobile-friendly)
  3. Rich preview of content being reviewed (not just "you have an approval")
  4. One-tap approve with optional feedback
  5. Structured reject with required reason

Escalation chain:
  - T+0: Assigned reviewer notified
  - T+X (configurable): Reminder sent
  - T+Y: Manager/backup notified
  - T+Z: Auto-action (reject, reassign, or escalate)

Smart scheduling:
  - Respect reviewer's working hours
  - Batch non-urgent approvals into daily digest
  - Flag urgent approvals with distinct notification style

Each layer has different persistence: - Brand Context: Updated by humans, versioned, shared across agents - Learning DB: Updated automatically from gate feedback, per-agent - Knowledge Graph: Updated via Cognee, shared across flows - Run Context: Exists only during flow execution, then archived

6. QA Steps

The Generator ≠ Grader Principle

The most important architectural decision in iEnable: the agent that creates content is NEVER the agent that evaluates it. This mirrors how every good creative organization works — the writer doesn't edit their own copy, the designer doesn't QA their own designs.

QA Node Configuration

qa_step:
  name: "Brand Compliance Check"
  evaluator_agent: "brand_qa_agent_v2"  # DIFFERENT from generator

  # What's being evaluated
  input: "{{product_copywriter.output}}"

  # Evaluation criteria (rubric)
  rubric:
    - criterion: "Brand Voice"
      description: "Does the copy match our warm, approachable tone?"
      weight: 0.3
      scoring: 1-10
      auto_fail_below: 4

    - criterion: "Factual Accuracy"
      description: "Are all product specs correct?"
      weight: 0.25
      scoring: pass/fail

    - criterion: "Platform Compliance"
      description: "Meets character limits, formatting rules for target platform?"
      weight: 0.2
      scoring: pass/fail

    - criterion: "SEO Quality"
      description: "Includes target keywords naturally?"
      weight: 0.15
      scoring: 1-10
      auto_fail_below: 5

    - criterion: "Differentiation"
      description: "Stands out from competitor copy?"
      weight: 0.1
      scoring: 1-10

  # Pass/fail logic
  passing:
    min_weighted_score: 7.0
    all_pass_fail_must_pass: true
    no_auto_fails: true

  # Output
  outputs:
    overall_score: number
    per_criterion_scores: object
    pass: boolean
    feedback: string  # detailed feedback for generator
    suggested_improvements: string[]

  # On fail
  on_fail:
    action: route_to_gate  # human reviews the QA failure
    # OR action: loop_back_to_generator  # auto-retry with feedback
    max_retries: 3
    include_feedback: true  # QA feedback sent to generator

QA Agent Independence

QA agents are: - Separate model instances — they don't share context with the generator - Separately prompted — their system prompt is about evaluation, not creation - Separately configured — different temperature (lower for QA), different model potentially - Separately learning — they learn what "good" looks like from human-approved outputs - Auditable — every evaluation is logged with full reasoning

QA → Gate Integration

The QA step feeds into a Gate:

Agent (Generate) → QA (Evaluate) → Gate (Decide)

If QA passes → Gate auto-passes (in AI QA mode)
If QA fails → Gate shows human the output + QA feedback
Human can:
  - Agree with QA → Reject (feedback to generator)
  - Override QA → Approve anyway (QA agent learns it was too strict)
  - Partially agree → Edit and approve (system learns the edit pattern)

This three-step pattern (Generate → QA → Gate) is iEnable's core content production primitive.

Learning Loop Mechanics

  1. Immediate: Rejection feedback is structured (taxonomy + detail) and stored
  2. Per-Run: If a loop/retry step exists, rejection feedback is injected into the next attempt's prompt
  3. Per-Agent: After every N decisions, lessons are extracted and added to the agent's long-term memory
  4. Per-Flow: Monthly, gate trust scores are recalculated. Promotion suggestions surface in the Approval Queue
  5. Per-Organization: Cross-flow patterns emerge. If "off_brand" is the #1 rejection reason across all flows, the brand bible may need updating — system alerts the admin

Execution Engine

Flow Run Lifecycle:

1. TRIGGER fires → creates a Run instance
2. Engine resolves first node(s) → executes
3. For each node:
   a. Resolve inputs (data mapping from upstream nodes)
   b. Execute (API call, agent inference, etc.)
   c. Store output in run context
   d. Determine next node(s) via edges
4. When a GATE is reached:
   a. Pause execution at this branch
   b. Create approval request
   c. Notify assignee(s)
   d. Wait for decision (async — engine continues other branches)
   e. On decision: resume or loop back
5. When all branches complete → Run finishes
6. Post-run: learning loop processes all gate decisions

Key architectural choices: - Event-driven, not polling-based — gates use webhooks/websockets, never polling - Parallel branch execution — branches run concurrently, Merge nodes sync them - Durable execution — runs survive server restarts (state persisted to DB) - Cost tracking — every API call's cost is recorded and attributed to the run - Audit log — every action, decision, and data transformation is logged

11. Pricing Implications

Based on competitive research, we recommend flow-run-based pricing (like n8n's execution model) rather than per-action (Make) or per-task (Zapier).
