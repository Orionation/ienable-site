1. Why This Exists

Jonathan asked Claude (via the Anthropic console) a simple question: "What would you do with these agents tonight?" Claude gave a thoughtful strategy for Tom and Adaline. Then Jonathan pasted it to me (Orion) and said: "Take his strategy and go harder/faster."

Claude's response ended with the most important line of the entire project:

"These report cards ARE the product. This is what iEnable sells."

That's the entire reason this system exists. We're not building dashboards for ourselves. We're building the product demo for iEnable — the thing that makes a CEO look at it and say "I need this for my company." When a potential customer sees Adaline's card showing $4.4M/year in proven ROI from a single AI recommendation, that's the sale.

The Problem We Were Solving

Before this system, our agents worked overnight and produced... markdown files. State reports. Learning databases. All trapped in local workspaces. Jonathan couldn't see what they were doing. Lyndsay (the ads director at Maxwood) couldn't see what Adaline found. The HODL group couldn't see Tom's research. The agents themselves couldn't see their own progress across sessions.

The agents were learning but nobody could SEE them learning. And if you can't see it, you can't sell it. And if you can't sell it, iEnable is just another AI tool nobody uses.

What Jonathan Kept Pushing For

Throughout the build, Jonathan's corrections shaped every decision:

  • "Slow down. You are acting instead of thinking." — I was patching links to static pages when he wanted me to make the dynamic pages LOOK like the static ones. The static pages had real content that was carefully crafted. The dynamic pages were bland.
  • "The static pages were for you to read and put that content into the database." — The static pages weren't the product. They were the SOURCE OF TRUTH. The data in them needed to live in the database so it could be served dynamically and updated every night.
  • "Look at what we did for Apollo — his card says what keywords he is targeting and his timeline to #1 on Google. That is the flywheel." — Every card must show the JOURNEY, not just tonight's output. Finding → Task → Execution → Measurement → Learning → Better Finding.
  • "The action plan and tasks should match." — If the card says "5 next steps" and the todo list shows different items, the system is lying. Plan items ARE tasks. Number them the same.
  • "This is a directory for THAT agent, not a nav to other agents." — When I built the directory card showing all agents, Jonathan corrected: each agent's directory should show only THEIR reports, grouped by type, expandable.
  • "Why would Tom's dashboard be dated? It is dynamic." — I was linking to a static dated page when the dynamic page is always current. Static pages are archives. Dynamic pages are the product.
  • "Does this put Tom on a path to $1M?" — The ultimate test. Every card, every task, every number should answer: is this agent moving toward its north star? If the card doesn't make you FEEL the progress, it's not done.

Why the ROI Tracker Changes Everything

Adaline recommended scaling GIF creative from 2% to 15% of ad spend. Lyndsay (the human) executed it — scaled to 13.1%. Result: +$369K/month ($4.4M/year). That's not a projection. That's measured. That's a closed loop: AI recommends → Human executes → Results measured → ROI proven.

When we show this to a potential iEnable customer, the conversation changes from "what can AI do?" to "how much money has it already made?" Every agent should have an ROI tracker card that answers that question.

Why Compounding Matters

Tom started with a hunch: "premium selling seems to work better than buying spreads." In 2 days, that became: 43 lessons → 6 hard rules from data → 3 validated learnings → 1 fully validated playbook (20/20 trades, 100% win rate across 8 market scenarios). That playbook now executes mechanically — Tom doesn't need to THINK about premium selling anymore, he just runs the playbook.

That compounding is invisible if it's trapped in a SQLite database on a laptop. The cards make it VISIBLE. The trajectory card shows the phases. The proof points show the evidence. The customer sees an agent getting smarter every single day.

As Jonathan said: "Every idle minute, the competition is catching up. This is winner-takes-all."

2. The Flywheel — Why It Has to Be a Loop

📋 Report Card
→
🎯 Tasks Created
→
🔄 Agent Executes
→
📊 Results Measured
→
📚 Learning Logged
→
📋 Better Report Card

Every cycle: findings become tasks → tasks produce results → results become learnings → learnings improve the next cycle

The reason this has to be a closed loop — not just a report or just a task list — is that AI agents have no persistent memory between sessions. Every time a cron fires and an agent wakes up, it's starting from scratch. The flywheel is how we solve that: the report card IS the agent's memory made visible. The tasks ARE the agent's intent made persistent. The learnings ARE the agent's intelligence made cumulative.

Without the loop, an agent could discover that DPA-Only campaigns are bleeding $10K/month... and then forget it next session. With the loop, that finding becomes a task ("Kill DPA-Only"), the task gets tracked until someone executes it, the execution gets measured ("saved $10K/month"), and the measurement becomes a validated learning that feeds every future decision.

The customer needs to SEE this loop. When someone looks at a card, they should feel: "This thing is getting smarter. It found something. Someone acted on it. There's proof it worked. And tonight it'll find more." That feeling is what sells iEnable.

The Learning Ladder

Knowledge compounds through levels. This matters because a "lesson" and a "playbook" are fundamentally different things:

  1. Observation — Raw finding. "Naked puts seem to win more often." Cheap to produce, easy to forget. Stored in learning.db.
  2. Hypothesis — Pattern with a testable prediction. "Premium selling has >80% win rate." Committed to paper. Worth tracking.
  3. Validated — Prediction confirmed by real outcome. "92% win rate on 12 trades." THIS IS GOLD. This is the agent proving it was right.
  4. Playbook — 3+ validations → permanent SOP. "20/20 trades, 100% win rate. Execute mechanically." This is the agent producing INSTITUTIONAL KNOWLEDGE that compounds forever.
Why Tom's Playbook Matters:
Tom went from observation → hypothesis → validated → playbook in 2 DAYS. That playbook is now a mechanical edge. He doesn't need to think about premium selling anymore — he just executes the rules. Every agent should be building playbooks like this. Adaline's "Bing scaling works at 3x spend" is close to playbook status. Muse's "background music adds 1-2 quality points" is validated. The cards need to show this progression because it's the PROOF that AI agents get better over time.

The ROI Loop — Why It's the Killer Feature

Most AI tools show you what they CAN do. This shows what they DID do — with numbers. The pattern:

  1. AI Recommends: "Scale GIF creative from 2% to 15%"
  2. Human Executes: Lyndsay scaled to 13.1%
  3. Results Measured: +$369K/month ($4.4M/year)
  4. Attribution Confirmed: High confidence — direct budget shift, channel-level measurement

This is tracked in the ROI Tracker card. Every agent should have one. The question it answers: "What has this agent made or saved me?" If you can't answer that with a number, the agent hasn't proven its value yet.

Jonathan's insight: Adaline's ROI tracker showing 2/2 recommendations executed with $4.4M proven impact is the single best sales tool for iEnable. A CEO sees that and the conversation is over — they want it.

Task Numbering

Tasks are automatically numbered to match the tonight_plan items in the report card. The frontend uses word overlap matching:

  1. For each plan item, find the task with the most matching words (minimum 2)
  2. Assign that task the plan item's number (1, 2, 3...)
  3. Unmatched tasks get numbers after the plan items (6, 7, 8...)
Best Practice: When creating tasks, use words from the tonight_plan item so the numbering auto-matches. E.g., plan says "Begin Bing Phase 2 scaling" → task name "Bing Phase 2 scaling roadmap for Lyndsay".

8. Card UI — The Frontend Template

Template Location

~/workspace/sites/agent-report/index.html

This single HTML file is deployed to ALL /r/{agent}/ directories. It reads the agent name from the URL path and fetches that agent's data from the API.

How It Determines the Agent

const pathParts = window.location.pathname.replace(/\/+$/,'').split('/');
const agentKey = pathParts[pathParts.length - 1].toLowerCase();

Data Fetching

On load (and every 60 seconds), it fetches three endpoints in parallel:

  1. /report-cards/latest — finds the entry matching agentKey
  2. /agent-status — gets live working/sleeping status
  3. /agent-tasks/active — filters tasks for this agent
  4. /report-cards?agent_name=X — historical cards for directory

Card Rendering Logic

The template has two rendering paths:

  1. Rich slides mode (if playbooks array has items with name/emoji/accomplishments) — renders each slide as a full card with its own header, stats, accomplishments, and key finding. Used by Adaline (7 slides) and Muse (5 slides).
  2. Flat mode (fallback) — renders a single Overview card + Next Steps card + optional Hard Rules/Validated/Hypotheses cards. Used by Tom, Apollo, iEnable, Orion.

In BOTH modes, the Trajectory card appears first (if trend data exists) and the Directory card appears last.

How Slides Were Created

The original rich cards were built as static HTML pages (e.g., /r/lyndsay/ for Adaline, /r/plankandbeam/muse/2026-02-28/ for Muse). We extracted the card data from those pages using Node.js to parse the JS const cards = [...] array, converted to JSON, and stored in the database via the playbooks field.

Extraction Process

# 1. Download the static page
curl -s https://ienable.ai/r/lyndsay/ > /tmp/lyndsay.html

# 2. Extract the cards JS array
python3 -c "
import re
html = open('/tmp/lyndsay.html').read()
match = re.search(r'const cards = (\[.*?\]);\s*\n', html, re.DOTALL)
with open('/tmp/extract.js', 'w') as f:
    f.write(f'const cards = {match.group(1)};\nconsole.log(JSON.stringify(cards));')
"

# 3. Parse with Node.js (handles JS syntax → JSON)
node /tmp/extract.js > /tmp/cards.json

# 4. Store in database
curl -X POST "$API/report-cards" -H "Content-Type: application/json" \
  -d "{\"agent_name\": \"Adaline\", ..., \"playbooks\": $(cat /tmp/cards.json)}"

10. Trajectory Card — The Journey

The first card on every agent page (when trend data exists). Shows:

  • North Star — the big goal in large bold text
  • Current Position — where they are now
  • Progress Bar — visual position through phases
  • Velocity Box (green) — how fast they're moving
  • Next Milestone Box (amber) — immediate target
  • Roadmap Timeline — vertical stepper with ✓ for done, green ring for current, gray for future
  • Proof Points — 4-column: Lessons, Validated, Hard Rules, Days Active

Phase Parsing

The trend.trajectory string is split on Phase \d+: regex. Each phase gets a class:

  • .done — phase before the one containing "(NOW)", or contains "(DONE)"
  • .current — contains "(NOW)"
  • No class — future phases

11. Directory Card

The last card on every agent page. Contains:

  • ← Back to Command Center link
  • 📋 Report Cards — expandable section, pulled from API history for that agent
  • 📊 Reports — expandable section, hardcoded per agent (static HTML reports deployed to FTP)

Report URLs (Hardcoded in Template)

The AGENT_REPORTS object in the template maps each agent to their static reports. When new reports are deployed, this object must be updated in the template and redeployed.

Future improvement: Store report URLs in the database (e.g., as a field on the report card) so the directory is fully dynamic and doesn't require template changes.

12. Nightly Loop — How Agents Compound

What Each Agent Does Every Night

  1. Read state-report.md — full context recovery document
  2. Read tonight-recap.md — what was planned
  3. Fetch task queue from API — GET /agent-tasks/active, filter by agent
  4. Pick highest-priority queued task, set to working
  5. Execute the work — research, create, analyze
  6. Log lessons to learning.db
  7. Feed insights to Cognee with domain tag
  8. Create new tasks for any opportunities discovered
  9. Update report card via API — new grade, accomplishments, stats, trend data
  10. Write tonight-recap.md — what was produced, learned, gaps, tomorrow's plan
  11. Write state-report.md — full recovery doc

The API Task Management Block

Added to each agent's nightly cron prompt on Feb 28, 2026:

=== API TASK MANAGEMENT (MANDATORY) ===
API: https://ienable-admin-app-hqtw2.ondigitalocean.app/api/v1

BEFORE starting work:
1. Fetch your tasks: curl -s "$API/agent-tasks/active" | jq '.data[] | select(.agent=="AGENT_KEY")'
2. Pick the highest-priority queued task and set it to working
3. Work on it. When done, create a result_summary with measurable outcome.

DURING work:
4. If you discover a new opportunity, CREATE a task for it.
5. Every finding = a task. Every task = a measurable outcome.

AFTER work:
6. Update your report card with tonight's results via POST /report-cards.
7. The trend field MUST include: north_star, current, velocity, milestone_next, days_active, trajectory.

YOUR NORTH STAR: [agent-specific goal]
Every session must move you closer.
Known Gap: There is no API endpoint to UPDATE task status. Agents can create tasks but cannot mark them working/complete via API. This needs to be added to the backend. Currently, task status changes require direct database access or a new endpoint.

# 1. Rebuild (embeds new-script.js into index.html)
python3 << 'EOF'
with open('sites/ienable-r/index.html', 'r') as f:
    lines = f.readlines()
with open('sites/ienable-r/new-script.js', 'r') as f:
    new_script = f.read()
script_start = script_end = None
for i, line in enumerate(lines):
    if '<script>' in line.strip() and script_start is None:
        script_start = i
    if '</script>' in line.strip() and script_start is not None:
        script_end = i
        break
new_lines = lines[:script_start] + [f'<script>\n{new_script}\n</script>\n'] + lines[script_end+1:]
with open('sites/ienable-r/index.html', 'w') as f:
    f.writelines(new_lines)
EOF

# 2. Deploy
lftp ... -e "cd public_html/ienable.ai/r; put sites/ienable-r/index.html -o index.html; bye"

16. Data Flow — End to End

How a Report Card Gets Created

  1. Agent runs nightly cron → reads state, executes work, logs lessons
  2. Agent (or Orion during morning audit) POSTs to /report-cards with structured data
  3. Database stores it with timestamp
  4. Frontend fetches /report-cards/latest on page load
  5. Template renders the card with trajectory, slides or flat mode, tasks, and directory

How Tasks Connect to Cards

  1. Report card has tonight_plan (5 numbered action items)
  2. Tasks are created in the API matching those plan items
  3. Frontend fetches tasks and auto-numbers them to match the plan
  4. The todo accordion on the card shows tasks with matching numbers
  5. When an agent completes a task, the result feeds the next report card

How Learnings Compound

  1. Agent writes to learning.db (SQLite) during nightly session
  2. Agent feeds Cognee knowledge graph with domain tag

17. Recovery Guide — When Things Break

Agent card shows "Failed to load"

  • Check API: curl -s https://ienable-admin-app-hqtw2.ondigitalocean.app/api/v1/report-cards/latest
  • If API is down, check DigitalOcean app status
  • If API returns data but page fails, check browser console for JS errors

Agent missing from command center

  • Verify report card exists: curl -s "$API/report-cards/latest" | jq '.data[] | select(.agent_name=="AgentName")'
  • If missing, POST a new report card for that agent

Slides not rendering (showing flat mode)

  • Check playbooks field: must be an array of objects with name, emoji, accomplishments keys
  • If playbooks has different structure (e.g., playbook metadata), it won't trigger slide mode

Task numbers don't match plan

  • Numbering is based on word overlap matching — ensure task names share at least 2 words with the plan item
  • If a task has no matching plan item, it gets numbered after the plan items

Need to rebuild from scratch

  1. Source files: ~/workspace/sites/agent-report/index.html (card template), ~/workspace/sites/ienable-r/ (command center)
  2. Database: All data in DigitalOcean PostgreSQL via the API
  3. Static pages: On FTP at eborro.com
  4. Backups: ~/workspace/backups/2026-02-28-1742-working-state/
